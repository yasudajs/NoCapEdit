# プロジェクト固有ルール

## 機能の実装作業の手順
- **実装作業の各ステップは、必ずユーザーの指示を受けてから行うこと。**（計画完了後に自己判断でコード変更などの実装ステップへ自動で進んではならない）
- 機能追加の際には、ユーザーとAIはファイルの修正等はおこなわず、修正内容についてディスカッションする。
- 修正内容の合意ができたら、AIは現在の開発対象バージョン（v0.1系なら `master`、v0.2系なら `v0.2`）から作業用ブランチを作成し、必ずspec.mdを最新版に更新してから、実装計画を作成すること。
- **バージョン番号の更新ルール**:
  - **v0.1系（`master` 派生）の場合**: バージョン管理ファイル（Cargo.toml 等の4ファイル）のバージョン番号をあらかじめ次の内部バージョン（`0.1.x`）に更新しておくこと。
  - **v0.2系（`v0.2` 派生）の場合**: バージョン管理ファイル（Cargo.toml 等の4ファイル）のバージョン番号をあらかじめ次の内部バージョン（`0.2.x`）に更新しておくこと。
- 作業に関わるドキュメントは、docs/に実ファイルとして逐次保存すること。
- ユーザーが実装計画書（および `task.md` などの作業用ドキュメント）を承認したら、AIはそれらのコミットおよびプッシュを行う。
- ユーザーの指示を受けて、AIは実装計画書に基づいて実装を行い、動作確認等の検証を行う。
- **実装および検証の完了後、ユーザーに確認を求める前に、`docs/history.md` に今回のバージョンの変更履歴を最上部に追記し、実装したソースコードおよび history.md の更新を含めてコミット＆プッシュする。**
- ユーザーが実装内容および変更履歴（docs/history.md）を承認したら、実装完了とし、クリーンアップ・マージ指示を待つ。

- **実装作業完了後のクリーンアップ・マージルール**:
  ユーザーからクリーンアップまたはマージの指示を受けたら、以下の通り作業ブランチの派生元に応じた処理を追加の確認指示を挟まずに一気に実施する（このフェーズは不要になったドキュメントやブランチの「削除・整理」のみを行う）：

  ### A. v0.1系（作業ブランチが `master` 派生の場合）
  1. `docs/history.md` に今回のバージョンの変更履歴が正しく追記されているか最終確認する。
  2. 作業関連のドキュメントフォルダ（`docs/[機能名]/`）を完全に削除してコミット＆プッシュする。
  3. `master` ブランチに切り替え、作業ブランチを非Fast-forwardマージ（`--no-ff`）してプッシュする。
  4. マージ完了後、ローカルおよびリモートの作業ブランチを削除する。

  ### B. v0.2系（作業ブランチが `v0.2` 派生の場合）
  1. `docs/history.md` に今回のバージョンの変更履歴が正しく追記されているか最終確認する。
  2. 作業関連のドキュメントフォルダ（`docs/[機能名]/`）を完全に削除してコミット＆プッシュする。
  3. `v0.2` ブランチに切り替え、作業ブランチを非Fast-forwardマージ（`--no-ff`）してプッシュする。**（※絶対に `master` にはマージしないこと）**
  4. マージ完了後、ローカルおよびリモートの作業ブランチを削除する。

- **`master`（v0.1）から `v0.2` への定期同期**:
  - ユーザーから `master` の変更を取り込む指示があった場合、または `master` にバグ修正等がマージされた際は、`v0.2` ブランチに切り替えて `git merge master` を実行し、`v0.2` を最新状態に保つこと。
- 本番リリース（0.1.35など）であれば、ユーザーが配布用のアーカイブファイルのビルドとGitHub CLIでのログイン（gh auth login）を完了したことを受けて、AIはGitHub Releaseへの登録およびREADME.mdの更新を行う。

## Gitの運用方針
- **コミットおよびプッシュのルール**:
  - **実装作業中のコミットおよびプッシュは、ユーザーの指示があったときのみ実施する**（未完成のドキュメントやバグを含む不要なコミット履歴を残さないため）。
  - コミットメッセージは日本語で記述する。
  - 1行目を20文字程度で作業内容の概要を記述し、その下に箇条書きで詳細を記述する。
- **マージのルール**:
  - 作業ブランチをマージする際は、ブランチでの開発履歴（分岐と合流）を視覚的に残すため、常に `git merge --no-ff <ブランチ名>` を使用して非Fast-forwardマージを行うこと。

## ドキュメントの運用方針
開発ドキュメント（実装計画書、タスク、ウォークスルー等）のライフサイクルは以下の通りとします。

1. **検討中・作業着手前 (WIP)**:
   - `docs/wip/[機能名]/` 配下にドキュメントを配置して検討を行います。このフォルダはクリーンアップ対象外です。
2. **実装開始時 (本番格上げ)**:
   - 実装が正式に開始されたら、対象のフォルダを `docs/wip/` から `docs/[機能名]/` (直下) に**移動**します（この時点で WIP からは削除されます）。
3. **実装完了時 (クリーンアップ)**:
   - 開発が終了し、ソースコードとドキュメントがGitにコミット＆プッシュされた後は、その機能の個別ドキュメントフォルダ（`docs/[機能名]/`）を**完全に削除**します。
   - 過去の履歴はGitのコミット履歴で管理するため、リポジトリ内に古いフォルダやファイルを残さないでください。

- **最新版のみの維持**:
  - 常に現在進行中の作業、または全体の共通設計に関する最新のドキュメントのみを残すようにしてください。
  - 機能追加するときには、実装計画書の作成前に `spec.md` を最新版に更新し、その後、実装計画を作成すること。

## リリース運用方針
- **内部バージョン**:
  - v0.1系: 変更ごとに `0.1.x` を1ずつインクリメント（小変更は統合可）
  - v0.2系: 変更ごとに `0.2.x` を1ずつインクリメント（小変更は統合可）
- **本番リリース**: 原則5ステップごと（v0.1系であれば 0.1.30, 0.1.35, 0.1.40...）にGitHub Releaseを作成
- **緊急修正**: 必要に応じて例外的に即リリース可
- **GitHub Release**: 本番リリース時のみ作成（内部バージョンではReleaseを作成しない）

## バージョン番号の管理ファイル
バージョンアップ時は、以下の**4ファイルを必ずセットで更新**すること。更新漏れはビルド成果物のバージョン不一致の原因になる。

| ファイル | 該当箇所 | 備考 |
|---|---|---|
| `Cargo.toml` | `version = "x.y.z"` | Rustビルドのバージョン管理 |
| `tauri.conf.json` | `"version": "x.y.z"` | Tauriバンドルのバージョン（アプリタイトルバー等に使用） |
| `nsis/installer.nsi` | `VERSION` / `VERSIONWITHBUILD` | NSISインストーラーのバージョン（`x.y.z.0` 形式） |
| `docs/DEVELOPMENT.md` | ZIPファイル名中のバージョン文字列 | ポータブル版ビルドコマンド例 |

以下は参照・記録用のため、バージョンアップ時の自動更新対象外（内容に応じて手動追記）:
- `docs/history.md` — 変更履歴の追記
- `README.md` — 本番リリース時のダウンロードリンク更新
