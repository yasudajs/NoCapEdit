# プロジェクト固有ルール

## 機能の実装作業の手順
- 機能追加の際には、ユーザーとAIはファイルの修正等はおこなわず、修正内容についてディスカッションする。
- 修正内容の合意ができたら、AIは作業用ブランチを作成し、必ずspec.mdを最新版に更新してから、実装計画を作成すること。
- 作業に関わるドキュメントは、docs/に実ファイルとして逐次保存すること。
- ユーザーが実装計画書（および `task.md` などの作業用ドキュメント）を承認したら、AIはそれらのコミットおよびプッシュを行う。
- ユーザーの指示を受けて、AIは実装計画書に基づいて実装を行い、ユーザーに作業結果の確認を求める。
- ユーザーが実装内容を承認したら、AIは修正したファイルのコミットおよびプッシュを行う。
- ユーザーが配布用のアーカイブファイルのビルドとGitHub CLIでのログイン（gh auth login）を完了したら、AIはGitHub Releaseへの登録およびREADME.mdの更新を行う。
- ユーザーの指示を受けて、AIは本番ブランチへのマージ（非Fast-forwardマージ）およびリリースバージョンのコミットとプッシュを行う。

## Gitの運用方針
- **コミットおよびプッシュのルール**:
  - コミットおよびプッシュをするときは、実行前に必ずユーザーに確認を行う。
  - コミットメッセージは日本語で記述する。
  - 1行目を20文字程度で作業内容の概要を記述し、その下に箇条書きで詳細を記述する。
- **マージのルール**:
  - 作業ブランチをマージする際は、ブランチでの開発履歴（分岐と合流）を視覚的に残すため、常に `git merge --no-ff <ブランチ名>` を使用して非Fast-forwardマージを行うこと。

## ドキュメントの運用方針
開発ドキュメント（実装計画書、タスク、ウォークスルー等）のライフサイクルは以下の通りとします。

1. **検討中・作業着手前 (WIP)**:
   - `docs/wip/[機能名]/` 配下にドキュメントを配置して検討を行います。このフォルダはクリーンアップ対象外です。
2. **実装開始時 (本番格上げ)**:
   - 実装が正式に開始されたら、対象のフォルダを `docs/wip/` から `docs/[機能名]/` (直下) に**移動**します（この時点で WIP からは削除されます）。
3. **実装完了時 (クリーンアップ)**:
   - 開発が終了し、ソースコードとドキュメントがGitにコミット＆プッシュされた後は、その機能の個別ドキュメントフォルダ（`docs/[機能名]/`）を**完全に削除**します。
   - 過去の履歴はGitのコミット履歴で管理するため、リポジトリ内に古いフォルダやファイルを残さないでください。

- **最新版のみの維持**:
  - 常に現在進行中の作業、または全体の共通設計に関する最新のドキュメントのみを残すようにしてください。
  - 機能追加するときには、実装計画書の作成前に `spec.md` を最新版に更新し、その後、実装計画を作成すること。

## リリース運用方針
- **内部バージョン**: 変更ごとに `0.1.x` を1ずつインクリメント（小変更は統合可）
- **本番リリース**: 原則5ステップごと（0.1.30, 0.1.35, 0.1.40...）にGitHub Releaseを作成
- **緊急修正**: 必要に応じて例外的に即リリース可
- **GitHub Release**: 本番リリース時のみ作成（内部バージョンではReleaseを作成しない）
