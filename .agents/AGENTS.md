# プロジェクト固有ルール

## 機能の実装作業の手順
- 機能追加の際には、ユーザーとAIはファイルの修正等はおこなわず、修正内容についてディスカッションする。
- 修正内容の合意ができたら、AIは作業用ブランチを作成し、必ずmvp-spec.mdを最新版に更新してから、実装計画を作成すること。
- 作業に関わるドキュメントは、docs/に実ファイルとして逐次保存すること。
- ユーザーが実装計画書（および `task.md` などの作業用ドキュメント）を承認したら、AIはそれらのコミットおよびプッシュを行う。
- ユーザーの指示を受けて、AIは実装計画書に基づいて実装を行い、ユーザーに作業結果の確認を求める。
- ユーザーが実装内容を承認したら、AIは修正したファイルのコミットおよびプッシュを行う。
- ユーザーが配布用のアーカイブファイルのビルドとGitHub CLIでのログイン（gh auth login）を完了したら、AIはGitHub Releaseへの登録およびREADME.mdの更新を行う。
- ユーザーの指示を受けて、AIは本番ブランチへのマージ（非Fast-forwardマージ）およびリリースバージョンのコミットとプッシュを行う。

## Gitの運用方針
- **コミットおよびプッシュのルール**:
  - コミットおよびプッシュをするときは、実行前に必ずユーザーに確認を行う。
  - コミットメッセージは日本語で記述する。
  - 1行目を20文字程度で作業内容の概要を記述し、その下に箇条書きで詳細を記述する。
- **マージのルール**:
  - 作業ブランチをマージする際は、ブランチでの開発履歴（分岐と合流）を視覚的に残すため、常に `git merge --no-ff <ブランチ名>` を使用して非Fast-forwardマージを行うこと。

## ドキュメントの運用方針
- **過去の機能ドキュメントのクリーンアップ**: 
  - 機能開発が終了し、ソースコードとドキュメントがGitにコミット＆プッシュされた後は、その機能の個別ドキュメントフォルダやファイル（`implementation_plan.md`、`task.md`、`walkthrough.md` 等）は削除してください。
  - 過去の作業履歴はGitのコミット履歴で管理するため、リポジトリ内にアーカイブ目的の古いフォルダやファイルを残さないでください。
- **`docs/wip/` フォルダはクリーンアップ対象外**:
  - `docs/wip/` 配下は「検討中・作業着手前」の機能ドキュメントを格納する場所であり、機能開発後のクリーンアップ対象には含めないこと。
  - `docs/wip/` 内の各サブフォルダは、対応する機能の実装が正式に開始されたタイミングで `docs/` 直下の作業フォルダに移動し、その後は通常のクリーンアップルールに従う。
- **最新版のみの維持**:
  - 常に現在進行中の作業、または全体の共通設計に関する最新のドキュメント（`implementation_plan.md` 等）のみを残すようにしてください。
  - 機能追加するときには、実装計画書の作成前に、mvp-spec.mdを最新版に更新し、その後、実装計画を作成すること。
