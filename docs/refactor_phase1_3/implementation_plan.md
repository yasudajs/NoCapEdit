# フェーズ 1-3: 正規表現処理の完全削除および連番ロジックの統一 (バックエンド)
*(旧題: 正規表現コンパイルのキャッシュ化)*

## 計画変更の経緯と理由

### 過去の実装計画（初期案）
当初のマスタープランでは、ファイルの移動やコピー時に同名ファイルとの衝突を回避するため、毎回実行されていた重い処理 `regex::Regex::new(r"_(\d{2})$")` を `std::sync::OnceLock` を用いてキャッシュ化し、パフォーマンスのオーバーヘッドを削減する計画でした。

### 問題点の発見
初期案に沿ってキャッシュ化の実装を行い手動テストを実施した結果、ユーザーから**「`abcd_01.nctx` というファイルをコピーすると `abcd_02.nctx` に統合されてしまい、元のファイル名が維持されない」**という問題が報告されました。
原因は、既存のコードが「ファイル名の末尾にある `_01` 等の連番を正規表現で強制的に切り落とし、元のファイル名を破壊した上で連番をインクリメントする」という仕様になっていたためです。

### 新たな計画に変更する理由
ユーザーが意図して `abcd_01` という名前を付けている場合、システムがそれを勝手に `abcd` の2番目のファイルとして `abcd_02` に変更するのは、文脈を破壊する不自然で危険な動作です。期待される自然な結果は、元のファイル名を維持したまま `abcd_01_1` のように派生することです。

この「元のファイル名を尊重する」という仕様を採用した場合、**そもそも「正規表現を用いて末尾を切り落とす処理」自体が完全に不要**になります。
したがって、「重い処理をキャッシュして高速化する」のではなく、**「重い処理そのものを完全に削除できる」**という、当初の目的を上回る究極の最適化につながるため、計画を変更しました。
あわせて、以前のフェーズで完了していた「同名ファイル回避時のゼロ埋め廃止と無限ループの上限厳格化（フェーズ1-2）」が移動・コピー関数には未適用だったため、この修正も統合し、アプリケーション全体の連番ロジックを一貫させます。

## 提案する変更 (新計画)

### バックエンド

#### [MODIFY] [main.rs](file:///c:/work/NoCapEdit/src/main.rs)

`move_file_or_dir` および `copy_file_or_dir` の2つの関数に対し、以下の修正を行います。

1. **正規表現処理の完全削除**
   初期案で実装した `OnceLock` キャッシュ化、および `Regex` を用いた末尾切り落とし（`re.replace`）のコードを完全に削除し、常に元のファイル名（`stem`）を連番付与のベースとして扱います。

2. **フェーズ1-2仕様への統合（ゼロ埋め廃止とループ上限）**
   連番付与のフォーマットをゼロ埋めなし（`_1` ～ `_9`）に変更し、無限ループによるクラッシュを防ぐため、探索回数が9を超えた場合はフェイルファストとしてエラーを返す安全機構を導入します。
