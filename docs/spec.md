## 2. プロダクト概要 & コアコンセプト
NoCapEdit は、起動直後からすぐに書き始められ、ファイル名を考えずに保存できることを重視したシンプルなテキストエディタである。

一般的なテキストエディタでは、新規作成直後はまだ保存先やファイル名が決まっていないため、保存時に別途入力が必要になる。NoCapEdit ではこの手間をなくし、起動時点で保存先とファイル名を自動的に確定させることで、書くことに集中できる体験を提供する。

### コア体験
- アプリを起動した瞬間に、新規テキストファイルが自動生成されて開いている
- ユーザーはすぐに入力を開始できる
- 入力停止時、タブ切替時、タブ終了時、アプリ終了時に自動保存される
- 複数タブで新規ファイルを扱える
- 保存失敗時に、最低限のリカバリ操作ができる

## 4. 機能仕様

### 4.1 初回起動とホームフォルダ
- **ホームフォルダ設定**:
  - 初回起動時にホームフォルダ設定画面を表示する。
  - デフォルトのホームフォルダは Windows の Documents 配下の nce フォルダとする。
    - 具体例: `C:/Users/<ユーザー名>/Documents/nce`
  - ユーザーはデフォルト値をそのまま採用することも、変更することもできる。
  - 設定したホームフォルダは `config.json` に永続化され、次回以降も自動的に使用される。

### 4.2 起動時・新規作成時のファイル自動生成
- **ファイル自動生成**:
  - アプリ起動時、ホームフォルダ配下に新規テキストファイル（.nctx）を自動作成する。
  - ファイル名形式は `yyyymmdd_hhmmss.nctx` （ローカル時刻）とする。
  - 生成されたファイルは作成後すぐにアクティブタブとして開く。
- **連番による衝突回避**:
  - 同名ファイルが既に存在する場合は、`_01`、`_02` のように連番を付けて回避する。

### 4.3 タブ管理と自動クリーンアップ
- **複数タブのサポート**:
  - 複数ファイルをタブで開け、タブ名にはファイル名を表示する。
  - タブバーにあるプラス（`+`）ボタンから新規ファイルを作成して新規タブとして開く。
  - プラス（`+`）ボタンはタブ領域のすぐ右側に配置され、タブの増加に合わせて右へ移動する。タブ領域が画面幅いっぱいに達した場合は、ボタン位置は右端で固定され、タブ領域のみが横スクロール可能となるUIを採用する。
  - 各タブは、ファイルパス、未保存フラグなどの内部状態を個別に保持する。
- **自動クリーンアップ（空ファイル削除）**:
  - タブを閉じる時、またはアプリを終了する時、対象タブのファイル名が自動生成規則（`yyyymmdd_hhmmss.nctx`）に一致し、かつ**内容が空または空白文字のみ**で構成される場合は、そのファイルをディスクから削除して閉じる。
  - **空白文字の定義**: 空文字、半角スペース、全角スペース（`\u3000`）、タブ、改行、およびそれらの組み合わせを空とみなす。
  - 上記の削除条件に当てはまらないタブについては、未保存フラグがある場合のみ保存を試行する。

### 4.4 テキスト編集と保存仕様（自動・手動・安全保存）
- **文字コード & 改行コード**:
  - 文字コードは UTF-8 とする。
  - 保存時の改行コードは Windows 向けとして **CRLF** に統一する。ファイル読み込み時は LF/CRLF を許容するが、保存時に正規化する。
- **自動保存**:
  - エディタ内のテキストが変更されたら未保存フラグを立てる。
  - 以下のタイミングで自動保存（未保存フラグがある場合のみ）を実行する。
    - ①入力停止から **3秒経過** 時（デバウンス処理）
    - ②タブ非アクティブ時（タブ切り替え時など）
    - ③タブを閉じる時
    - ④アプリ終了時
- **手動保存 (`Ctrl + S`)**:
  - `Ctrl + S` キー押下時、3秒の待機時間を待たずに、即座にアクティブタブのテキスト内容を上書き保存する。
  - 手動保存が実行された時点で、進行中だった自動保存タイマーはクリア（キャンセル）される。
- **安全保存方式**:
  - 保存時にファイル破損を防ぐため、一時ファイルに書き出してから置換する方式を採用する。これにより OneDrive 等のクラウド同期フォルダでの競合を防ぐ。
- **保存失敗時のリカバリ**:
  - 保存失敗時はステータスバーやダイアログでユーザーに通知する。
  - アプリ終了時の保存失敗では「再試行」「別名で保存」「終了を中止」の選択肢を提示する。

### 4.5 画面表示（タイトルバー・ステータスバー）
- **タイトルバーバージョン表示**:
  - ウィンドウタイトルに `NoCapEdit [ Ver X.Y.Z ]` を表示する。
  - バージョン番号は `Cargo.toml` の `version` を一元的なマスター（Source of Truth）とし、バックエンド経由でフロントエンド（`document.title`）に動的に反映する。
- **保存状態のステータス表示**:
  - 画面下部のステータスバーに、保存状態（`保存済み` / `編集中` / `保存中...` / `保存失敗`）を表示する。
  - ユーザーの集中を妨げないため、ポップアップ（トースト）通知は表示せず、ステータスバーのテキスト更新のみで静かに通知する。
- **エディタ情報表示**:
  - ステータスバーの右側には、現在のカーソル位置、文字数、フォントサイズ、行間高さを表示する。
  - 文字数のカウントは、改行文字（`\r` および `\n`）を含まない純粋な文字数とする。
  - 表示形式: `Ln X, Col Y | Z chars | Font xx pt | LH x xx`

### 4.6 独自拡張子（.nctx）とOS関連付け・単一ウィンドウ動作（シングルインスタンス）
- **ファイル保存と拡張子**:
  - ファイル保存（別名保存）ダイアログでは、デフォルトを `.nctx` (NoCap Text) としつつ、`.txt`（一般テキスト）、`.ncmd`（Markdown）も選択可能とする。
- **OSファイル関連付け**:
  - インストール時に `.nctx` 拡張子をアプリに関連付け、OS上でダブルクリックした際に NoCapEdit で開けるようにする。
- **単一ウィンドウ動作（シングルインスタンス）**:
  - アプリは常に単一のメインウィンドウでのみ動作する（多重起動の防止）。
  - すでにアプリが起動している状態で、新たに `.nctx` ダブルクリック等により別プロセスで起動された場合、新規プロセスは起動引数のファイルパスを既存ウィンドウに通知して終了する。
  - 既存ウィンドウは通知されたファイルパスを受け取り、新規タブとして開きアクティブタブとする。また、ウィンドウが最小化や背後にある場合は復元し、最前面にフォーカスする。

### 4.7 ユーザー設定と永続化
ユーザー設定は設定ファイル（`config.json`）に保存され、次回起動時も自動的に適用される。

- **テーマ切り替え**:
  - ライトモードとダークモード（完全な黒）をサポートし、太陽/月アイコンのクリックで切り替える。
  - DWM APIを利用し、ダークモード時はWindowsのウィンドウタイトルバーも完全な黒（`#000000`）に同期させる。
- **フォントサイズ調整 (ズーム)**:
  - 最小 `8pt`、最大 `72pt`、変更ステップ `1pt`。
  - トリガー1: `Ctrl` + マウスホイール回転
  - トリガー2: `Ctrl` + `+`（拡大） / `-`（縮小） キー
- **行間高さ調整**:
  - 最小 `1.0`、最大 `3.0`、変更ステップ `0.1`（画面上の表記は `pt` を付与）。
  - トリガー1: `Ctrl` + `Shift` + マウスホイール回転
  - トリガー2: `Ctrl` + `Shift` + `+` / `-` キー
- **フォントファミリー選択 (遅延ロード搭載)**:
  - エディタのフォントファミリーを等幅（Monospace）とその他に分類してセレクトボックスから選択できる。
  - **遅延ロード**: 起動時にはシステムフォントの全検出を行わず、設定済みの適用中フォントのみを一時表示して起動を高速化する。ユーザーがセレクトボックスをクリックした瞬間に非同期でシステムフォントの検出を開始し、全リストに更新する。
- **インデント設定 & タブ入力制御**:
  - エディタ上で `Tab` キーを押下した際、他のフォーカス移動を行わず、インデント文字を挿入する。
  - 設定ダイアログから「タブ文字（`\t`）」「スペース2文字」「スペース4文字」の挙動を選択できる。
  - 複数行選択した状態で `Tab` を押下した際、選択範囲のすべての行頭にインデント文字を追加する。
  - `Shift + Tab` 押下時、選択範囲の各行頭からインデント文字を1段階分削除する。


### 4.8 設定値・定数の一元管理（ハイブリッドアプローチ）
アプリケーション全体のマジックナンバーや設定値を一元管理し、保守性を高めるために以下の役割分担（ハイブリッドアプローチ）を採用する。
- **基本設定・振る舞い（Rust側で管理）**: アプリケーションの基本定数（デフォルトの保存先パス、初期テーマ設定、ファイル拡張子、ウィンドウのデフォルトサイズ、プロセス間通信のポート番号など）は、Rust側の定数モジュール（`constants.rs` 等）を「真なる情報源（Single Source of Truth）」として一元管理し、フロントエンドは起動時にTauriのAPI経由で値を取得する。
- **デザイン・UI設定（CSS変数で管理）**: フォントファミリの定義文字列（`Fira Code...` 等）やサイズ、テーマカラーなどの描画に関する値は `style.css` の `:root` に集約し、JSからはCSS変数を書き換えることでデザインを変更する設計とする。
- **フロントエンド固有のロジック（JSで管理）**: オートセーブの遅延時間（`3000ms`）やDOM要素のIDなど、OSやバックエンドに無関係なブラウザ固有の定数は、JS側の定数定義（または `config.js`）に集約する。

### 4.9 自動アップデートチェック機能
- **バックグラウンドチェック**:
  - アプリ起動直後のUI描画や初期化を妨げないよう、数秒の遅延（例：3秒程度）を挟んでからバックグラウンドで最新バージョン情報の取得を行う。
  - GitHub API（ReleasesのLatestエンドポイント）を利用して最新バージョンを取得する。
  - ネットワークエラーやAPI制限等による取得失敗時は、エラーを握りつぶしユーザーへの通知は一切行わない。
- **通知UIの表示**:
  - 新しいバージョンが存在する場合、以下の2箇所に控えめな通知を表示する。
  1. **タイトルバー**: ウィンドウタイトルに `NoCapEdit [ Ver X.Y.Z ] (Update: vA.B.C)` の形式で追記する。
  2. **設定画面の最上段**: 設定ダイアログ内の最上部に、アップデート可能な旨（`アップデート可能です Ver. X.Y.Z → Ver. A.B.C`）とリリースノートへのリンクを表示する。リンククリック時は Tauri の `shell.open` API によりシステムの標準ブラウザで安全に外部ページを開く。
