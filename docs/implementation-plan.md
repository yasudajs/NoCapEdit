# NoCapEdit 実装計画書

## 1. 文書情報
- アプリ名: NoCapEdit
- 対象: MVP仕様書に基づく初期バージョン実装
- 対象OS: Windows
- 開発技術: Tauri + Rust
- 作成日: 2026-07-04

## 2. 開発環境と前提条件
### 2.1 必須環境
- Rust 1.70 以上
- Node.js 16 以上
- npm または yarn
- Visual Studio Build Tools（Windows）
- Tauri CLI

### 2.2 セットアップ手順
```bash
# Tauri CLI インストール
npm install -g @tauri-apps/cli

# プロジェクトフォルダで依存関係をインストール
npm install
cargo build
```

### 2.3 初期プロジェクト構成
- Tauri スケルトン生成済み（既に Cargo.toml で管理）
- フロントエンド: HTML/CSS/JavaScript（Tauri 標準）
- バックエンド: Rust（Tauri invokable commands）

## 4. 実装タスク分解と依存関係

### Phase 1: 基盤構築（1-2日）

**受け入れ基準:**
- アプリケーション起動でウィンドウが表示される
- 画面上に「タブバー」「編集エリア」「ステータスバー」の領域が見える
- スタイルが Obsidian 風で落ち着いた見た目

**実装詳細:**
- Tauri イベントリスナー設定
- CSS Grid または Flexbox でレイアウト実装
- ダークカラースキーム定義

#### Task 1-2: ホームフォルダ設定の永続化
**所要時間:** 2-3時間
**依存関係:** Task 1-1

- 設定ファイル保存先の定義（`%APPDATA%/NoCapEdit/config.json` など）

**受け入れ基準:**
- 初回起動時にホームフォルダ設定ダイアログが表示される
- デフォルト値 `C:/Users/<username>/Documents/nce` が表示される
- ユーザーが変更したホームフォルダが config.json に保存される
- 2回目以降は保存済み値が使用される

**実装詳細:**
- `tauri::fs` モジュールでファイル操作
- JSON シリアライゼーション（serde + serde_json）
- 初回起動判定（設定ファイルの有無確認）

### Phase 2: ファイル操作の基盤（1-2日）
**目的:** ファイル自動作成、読み書き、タブ管理

#### Task 2-1: ファイル自動作成とタイムスタンプ採番
**所要時間:** 2-3時間
**依存関係:** Task 1-2
**成果物:**
- ファイル名生成ロジック（yyyymmdd_hhmmss.txt）
- 連番衝突回避機構
- ホームフォルダ自動作成

**受け入れ基準:**
- アプリ起動時にホームフォルダ配下にファイルが自動作成される
- ファイル名が `yyyymmdd_hhmmss.txt` 形式である
- 同名ファイルが存在したら `_01`, `_02` で回避される
- ホームフォルダがなければ自動作成される

**実装詳細:**
- `chrono` クレートで時刻取得
- パス衝突チェックループ
- ファイル権限チェック

#### Task 2-2: タブ管理の内部状態定義
**所要時間:** 2-3時間
**依存関係:** Task 1-1, Task 2-1
**成果物:**
- TabState 構造体定義
  - file_path
  - is_auto_created
  - created_in_current_session
  - is_dirty
  - has_non_whitespace_input
- タブ追加・削除・切替ロジック

**受け入れ基準:**
- メモリ上でタブの状態を保持できる
- タブ追加時に ID が自動採番される
- アクティブタブ切替が可能
- デバッグ出力で各フラグ値が確認できる

**実装詳細:**
- `uuid` クレートでタブ ID 生成
- Arc<Mutex<>> で thread-safe な状態管理
- Tauri state で管理

#### Task 2-3: ファイルの読み書き基本実装
**所要時間:** 2-3時間
**依存関係:** Task 2-1, Task 2-2
**成果物:**
- ファイル読み込み関数
- ファイル上書き保存関数
- 一時ファイル方式での安全保存

**受け入れ基準:**
- ファイルが作成後すぐに読み込める
- テキスト編集後にファイルが上書き保存される
- 保存中にクラッシュしてもファイル破損しない（.tmp ファイル使用）
- 保存失敗が検出できる

**実装詳細:**
- `std::fs` または `tokio::fs` で非同期ファイル操作
- 一時ファイル → 置換パターン実装
- エラーハンドリング

### Phase 4: 自動保存機能（2-3日）
**目的:** 自動保存とイベント駆動の実装

#### Task 4-1: Debounce 自動保存タイマー
**所要時間:** 2-3時間
**依存関係:** Task 2-3, Task 3-2
**成果物:**
- キーボード入力検出
- 3秒のタイマーロジック（debounce）
- タイマーのリセット機構

**受け入れ基準:**
- 入力停止から 3 秒後に自動保存が実行される
- 入力中はタイマーがリセットされ続ける
- 無操作時に延々とポーリングが続かない（イベント駆動）

**実装詳細:**
- JavaScript: `setTimeout` + `clearTimeout`
- または Rust 側で `tokio::time::interval` を選択可能
- キーイベント → Rust invoke command → タイマー起動パターン

#### Task 4-2: タブ切替時の保存
**所要時間:** 1-2時間
**依存関係:** Task 4-1
**成果物:**
- タブ切替イベント検出
- 前タブの is_dirty チェック
- 前タブの保存実行

**受け入れ基準:**
- タブ切替時に未保存内容が自動保存される
- 保存中に他タブへ切り替えてもエラーが出ない

**実装詳細:**
- タブクリックイベント → 前タブ保存チェック → 新タブ表示
- Rust 側で is_dirty フラグ確認後保存

#### Task 4-3: タブクローズ時の保存と削除判定
**所要時間:** 2-3時間
**依存関係:** Task 4-2, Task 2-2
**成果物:**
- タブクローズ前の保存判定
- 空白のみファイルの削除ロジック
  - is_auto_created = true
  - created_in_current_session = true
  - content.trim() が空
  - has_non_whitespace_input = false
- ファイル削除実行

**受け入れ基準:**
- 何も入力していないタブを閉じたらファイルが削除される
- 1文字でも入力したタブはファイルが残される
- 既存ファイルは削除されない

**実装詳細:**
- content trim 判定（空文字、スペース、タブ、改行のみ）
- フラグチェック
- ファイル削除前に確認ロジック

#### Task 4-4: アプリ終了時の保存と削除
**所要時間:** 2-3時間
**依存関係:** Task 4-3
**成果物:**
- 終了前イベントハンドリング
- 全タブ走査と保存/削除判定
- 保存失敗時のリカバリーダイアログ

**受け入れ基準:**
- アプリ終了時に全てのタブが処理される
- 空白のみファイルは削除される
- 保存必要なファイルは保存される
- 保存失敗時に「再試行」「別名保存」「終了中止」が選べる

**実装詳細:**
- Tauri の close_requested イベント使用
- 非同期処理で全タブ保存
- Promise.all で並列処理（複数タブ対応）

## 7. 技術スタック詳細

### Rust クレート推奨
```toml

serde_json = "1.0"
chrono = "0.4"

regex = "1.0"
```

### JavaScript ライブラリ推奨
- **エディタ:** 初期は <textarea>、将来 CodeMirror へ
- **ダイアログ:** Tauri 組み込み API
- **通知:** iziToast または Tauri notify
- **UI Framework:** 不要（vanilla JS + CSS Grid）

## 9. リスクと対策

### リスク 1: Tauri フロントエンド・バックエンド間通信の遅延
**対策:** イベント駆動とコマンド非同期化で対応

### リスク 2: ファイル保存中のクラッシュ
**対策:** 一時ファイル → 置換方式で対応

### リスク 3: 大量タブでのメモリ不足
**対策:** MVP段階ではタブ数制限は入れない。必要に応じ後で対応

### リスク 4: Windows のファイルロック問題
**対策:** ファイルを開く際の排他制御を検討
