# [実装計画書] 連番ファイル名正規表現の不一致修正・一元化および手動保存ロジックの共通化 (1-1, 1-2)

## 概要
`spec.md` (セクション 4.2, 4.3, 4.4) に定義されている自動生成ファイル名形式 (`yyyymmdd_hhmmss.nctx`) および連番形式 (`_1`〜`_9`) と、フロントエンド (JS) 側の実装の相違・不具合を修正します。

1. **1-1. 連番正規表現の不一致修正とDRY化**:
   `AUTO_FILE_REGEX` 等が `_\d{2}` (2桁想定) となっていたため、連番付きファイル (`_1` 等) が自動生成ファイルと判断されず、タブ表示名フォーマットの不全や自動クリーンアップ処理の漏れが発生していた問題を修正します。
2. **1-2. 手動保存ロジックの共通化 (DRY化) と連番回避の適用**:
   手動保存モード (`saveMode === 'manual'`) 時に `Ctrl + S` を押した際、独自のハードコーディング処理が実行され連番衝突回避や既存ファイル上書きが正常に行われていなかった不具合を修正し、共通保存関数 `saveTabIfDirty()` へ処理を一元化します。

## ユーザー確認済み事項
- **正規表現の連番パターン**: バックエンド実装および仕様（最大 `_9`）と厳密に合わせ、1桁 (`1`〜`9`) のパターン (`(?:_([1-9]))?` または `(?:_(\d))?`) を採用。
- **重複定義の解消 (DRY化)**: `state.js` と `helpers.js` に存在した `AUTO_FILE_REGEX` を `state.js` に一元管理し、共通インポート化する。
- **手動保存ロジックのDRY化**: `triggerManualSave()` における直書き処理を廃止し、`saveTabIfDirty()` を呼び出すことで、手動保存時にも連番衝突回避・正常な上書き保存を適用する。

## 検証計画

### 手動検証
1. **タブ表示名フォーマット検証**:
   - `yyyymmdd_hhmmss_1.nctx` という連番付きファイルを作成/保存、または既存ファイルとして開き、タブ表示名が `yyyy/mm/dd hh:mm:ss-1` と正しくフォーマットされることを確認。
   - 手動保存モード時、`[yyyy/mm/dd hh:mm:ss-1]` と表示されることを確認。
2. **手動保存モードでの連番衝突回避・上書き検証**:
   - 手動保存モードで同秒数内に `Ctrl + S` を連打/複数保存した際、連番 (`_1`等) が付加され既存ファイルが破壊・上書きされないことを確認。
   - 保存済みファイルに対して追記・編集して `Ctrl + S` を押した際、新たなタイムスタンプファイルが乱立せず既存ファイルに正しく上書き保存されることを確認。
3. **自動クリーンアップ検証**:
   - 同秒数で新規ファイルを作成し、`_1` 付きの自動生成ファイル（実体あり）を作成させた状態で内容を空にしてタブを閉じた時、ファイルが自動削除されるか検証。
