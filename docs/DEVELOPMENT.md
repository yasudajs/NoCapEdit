# 🛠️ 動作環境とビルド・開発方法

現在のバージョン は Windows 環境を対象として構成されています。

### 前提条件
- Rust 1.70 以上
- Visual Studio Build Tools (Windows)
- Node.js (フロントエンドのビルドおよびパッケージ管理)
- Tauri CLI (`cargo install tauri-cli`)

### 開発・起動手順
ターミナル（PowerShell等）でプロジェクトルートディレクトリを開き、以下のコマンドを実行します。

```powershell
# アプリケーションのビルドと起動
cargo run
```
*(※ Tauri CLIがインストールされている場合は `cargo tauri dev` でも起動可能です)*

### 配布パッケージのビルド
インストーラーをビルドする場合は以下のコマンドを実行します。

```powershell
cargo tauri build
```

また、ポータブル版（ZIP形式）を作成する場合は、以下のコマンドを実行して `target/release/bundle` 配下に ZIP アーカイブを出力します。

```powershell
# ポータブル版のビルドとZIPアーカイブの作成
# (Windowsアイコンキャッシュの競合回避のため、実行ファイル名を「NoCapEdit_v02.exe」にリネームしてZIP化します)
cargo build --release
New-Item -ItemType Directory -Force -Path "target/release/bundle"
Rename-Item -Path "target/release/NoCapEdit.exe" -NewName "NoCapEdit_v02.exe"
Compress-Archive -Path "target/release/NoCapEdit_v02.exe" -DestinationPath "target/release/bundle/NoCapEdit-0.2.28-Portable.zip" -Force
Remove-Item "target/release/NoCapEdit_v02.exe"
```

## 💡 開発ノウハウ・トラブルシューティング

### 将来の多言語化（i18n）に向けた準備方針
現在、将来的なMicrosoft Store配信やクロスプラットフォーム（Mac/Linux）展開、および多言語化を見据え、フロントエンドのUIテキストを `src/dist/i18n.js` に集約する方針としています。
- 新しくJS内で動的に表示するテキスト（エラー表示やダイアログ）を追加する場合は、ハードコードを避け、`i18n.js` 側へキーと日本語文言を登録し、`t('キー名')` 経由で呼び出してください。

### Windows環境におけるファイル監視とパスの取り扱いに関する注意点
Windows上で `notify` クレートを用いたファイル監視を実装する際、以下の仕様による影響を考慮する必要があります。

1. **Rustの `canonicalize()` によるUNCプレフィックス付与**
   Rustの `std::fs::canonicalize` をWindowsで実行すると、長すぎるパスに対応するため自動的に `\\?\` というUNCプレフィックスが付与されます。一方で `notify` から通知されるイベントのパスには通常このプレフィックスは付いていません。そのため、フロントエンド（JavaScript）側で単純に文字列比較を行うと一致しないという問題が発生します。
   - **解決策**: フロントエンド側でのパス比較の前に、必ず正規化処理（`\\?\` の除去、バックスラッシュのスラッシュ化、末尾スラッシュの除去、小文字化）を統一して行う必要があります。

2. **上書き保存時の `Remove` イベント発火**
   `notify` で監視しているフォルダ内でファイルを上書き保存する際、古いファイルを `fs::remove_file` してから一時ファイルを `fs::rename` で配置すると、当然ながら「ファイルが削除された」という `Remove` イベントが発火します。さらにWindowsの仕様により、`remove_file` を省いて `fs::rename` だけで既存ファイルへ上書きした場合でも、OSレベルで置換処理が行われる影響で対象ファイルに対する `Remove(Any)` イベントが発火してしまいます。
   - **解決策**: 上書き保存時には一時ファイル（`tmp`）を用いたアトミックなリネーム保存ではなく、対象ファイルへ直接 `fs::write` を行うことで、意図しない削除イベントの発火（およびそれに伴うフロントエンドでのタブ自動クローズなどの誤動作）を防ぐことができます。

### ⚠️ マルチプラットフォーム化に向けた既知の制限事項
現在（Ver 0.2系）の実装では、Windows環境特有のパス表記揺れを吸収するため、フロントエンド側でパス文字列をすべて小文字化（`toLowerCase()`）してから比較する処理を行っています。
しかし、将来的にLinux等（ファイル名の大文字小文字を厳密に区別するファイルシステム）へ展開する際には、この実装のままでは `aaa` と `AAA` が同一視されてしまう重大なバグの原因となります。
- **今後の課題**: マルチプラットフォーム対応を行う際は、実行環境のOSを判定して比較ロジックを切り替えるか、Rustのバックエンド側でパスを一意に正規化してフロントエンドへ渡す設計への見直しが必要となります。

### Windows環境におけるTauri WebView内のドラッグ＆ドロップ（D&D）強制キャンセル問題
Tauri（特にWindows環境）において、HTML5 Drag & Drop を用いた内部要素のドラッグを実装する際、ドラッグを開始した瞬間に即座に `dragend` が発火してドラッグがキャンセルされてしまう不具合が発生します。

1. **Tauriのファイルドロップ機能（fileDropEnabled）との衝突**
   - **原因**: Tauriのデフォルト機能である外部ファイルドロップの受け入れハンドラが、WebView内部のドラッグイベントをインターセプトしてしまい、操作を強制終了させてしまいます。
   - **対策**: Rust側のメインウィンドウ生成（`WindowBuilder`）時に、`.disable_file_drop_handler()` メソッドを明示的に呼び出してデフォルトのドロップハンドラを無効化してください。

2. **Windows OLE ドラッグ時のUNCパス（`\\?\`）の影響とドラッグプレビューエラー**
   - **原因**: `e.dataTransfer.setData` に `\\?\` で始まる拡張パス（UNC）を設定すると、Windowsのドラッグマネージャーが解釈できずにドラッグを拒否します。また、複雑なDOM構造を持つ要素のドラッグプレビューイメージの生成失敗によっても即座にキャンセルされることがあります。
   - **対策**: 
     - ドラッグ時に設定するパスから正規表現等で `\\?\` プレフィックスを除去した標準のパス（`C:\...`）を渡してください。
     - `e.dataTransfer.setDragImage` に 1x1 の透明GIF画像をダミー画像として設定するハックを適用し、プレビューイメージ生成時のエラーを回避してください。

### エディタの自動フォーカスとツリー操作のキーイベント（Delete/Shift+Delete）競合問題
ファイルを新規に開いた際、すぐに文字入力を開始できるようにエディタ（`textarea`）に自動でフォーカス（`focus()`）を奪う仕様にした結果、ツリー側で `Delete` 等のキーボード操作がエディタ入力中と見なされ、削除が動作しなくなる不具合が発生しました。

1. **ツリー側のフォーカス管理**
   - **対策**: `div` 要素などのツリー項目には標準でフォーカスが当たらないため、`tabindex="0"` を付与してフォーカス可能にし、ツリーの項目がクリック・右クリックされた瞬間に `.focus()` を走らせてフォーカスをツリー側に引き戻す必要があります。

2. **すでにアクティブなファイルクリック時の処理**
   - **対策**: すでにエディタで開いて現在アクティブになっているファイルをツリーでクリックした際は、ファイルの再ロード処理（およびエディタへのフォーカス強制遷移）をスキップし、フォーカスをツリー上に留めます。これにより、開いているファイルを「クリック → Deleteキーで削除」する操作が可能になります。

3. **見た目とフォーカス状態の一致**
   - **対策**: エディタがフォーカスを得た（`focus`）イベントを検知し、ツリーの選択状態を「アクティブ（青色）」から自動的に「非アクティブ表示（グレー）」に切り替えることで、フォーカス状態と画面表示の矛盾を解消しています。

### ファイルツリー自動更新時におけるスクロール位置・展開・選択状態の維持（差分更新設計）
自動保存や外部ファイル監視による自動更新、または手動リロードが発生した際、ファイルツリーのDOMを丸ごと再生成（`innerHTML = ''`）すると、スクロール位置がリセットされたり、展開していたフォルダが閉じたり、選択状態が消失してしまいます。

これを防ぐため、本アプリでは以下のインクリメンタルな差分更新アルゴリズムと設計を採用しています：

1. **既存DOMの再利用とインクリメンタル同期**
   - フォルダが読み込まれるたびにツリーをクリアするのではなく、現在表示されているDOM要素（`.tree-item` など）の集合と、最新のファイルシステムデータのリストを突き合わせます。
   - データから消えたファイルに対応するDOMのみを削除し、データに新しく現れたファイルのみをヘルパー関数 `createTreeItemElement` で生成して追加します。

2. **DOMの順序変更（Reorder）時のレイアウトシフト防止**
   - ファイル名の変更やソート順の変更に伴いDOMの順序を整列する際、すべてのDOM要素に対して一律に `appendChild` などを呼ぶと、既存のDOM要素が一時的にデタッチ（取り外し）されてから再配置されます。ブラウザはこの動作を「レイアウトの大きな変化」と解釈するため、スクロール位置（`scrollTop`）が `0` にリセットされます。
   - **解決策**: 現在のDOMの子要素の順序と、最新データの順序を比較しながらループ処理します。すでに正しい並びになっているDOMについては移動処理（`appendChild` / `insertBefore`）をスキップし、順序が異なっている要素のみ最小限の `insertBefore` を行うことで、スクロール位置と展開状態を完全に維持します。

3. **選択状態（`selected`）およびフォーカスの追従**
   - ファイルの作成・名前変更や移動（ドラッグ＆ドロップ、切り取り貼り付け）の際、動作したファイルのパス（`selectedPath`）やDOM要素（`selectedElement`）をトラッキングしています。
   - DOMの更新が終わった後に最新のDOMツリーから `selectedPath` と一致する要素を走査し、`selectedElement` を上書き更新して選択ハイライト（青/グレー）やフォーカスを当てることで、ユーザーのキーボード操作や表示状態が維持されます。
