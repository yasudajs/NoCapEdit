# NoCapEdit MVP仕様書

## 1. 文書情報
- アプリ名: NoCapEdit
- 略称: nce
- 文書目的: 初期版の実装範囲と判断基準を固定する
- 対象バージョン: Windows向け初期MVP
- 作成日: 2026-07-04

## 2. プロダクト概要
NoCapEdit は、起動直後からすぐに書き始められ、ファイル名を考えずに保存できることを重視したシンプルなテキストエディタである。

一般的なテキストエディタでは、新規作成直後はまだ保存先やファイル名が決まっていないため、保存時に別途入力が必要になる。NoCapEdit ではこの手間をなくし、起動時点で保存先とファイル名を自動的に確定させることで、書くことに集中できる体験を提供する。

## 3. MVPの目的
本MVPの目的は、以下の体験をWindows上で成立させることである。
- アプリを起動した瞬間に、新規テキストファイルが自動生成されて開いている
- ユーザーはすぐに入力を開始できる
- 入力停止時、タブ切替時、タブ終了時、アプリ終了時に自動保存される
- 複数タブで新規ファイルを扱える
- 保存失敗時に、最低限のリカバリ操作ができる

## 4. 対象プラットフォームと技術方針
### 4.1 対象プラットフォーム
- 初期MVPは Windows を対象とする
- 将来的に macOS、Linux、CLI 版への展開を視野に入れる

### 4.2 開発技術
- アプリケーション基盤: Tauri
- コア実装言語: Rust
- UI方針: モダンで落ち着いた見た目を目指し、Obsidian や Zed のような雰囲気を参考にする

## 5. 想定ユーザーと利用シーン
### 5.1 想定ユーザー
- 思いついた内容をすぐに書き留めたい個人ユーザー
- ファイル名を考える操作を面倒に感じるユーザー
- シンプルな編集体験を好むユーザー

### 5.2 主な利用シーン
- メモの書き出し
- 作業ログの記録
- 下書きテキストの一時作成

## 6. MVPの対象範囲
### 6.1 MVPに含める機能
- 初回起動時のホームフォルダ設定
- 起動時の新規ファイル自動作成
- タブによる複数ファイル編集
- タブ上のプラス操作による新規ファイル作成
- 自動保存
- 終了前保存
- 保存失敗時の再試行、別名保存、終了中止
- 保存状態のステータス表示

### 6.2 MVPに含めない機能
- 既存ファイルを開く機能
- 高度な設定画面
- 検索、置換
- シンタックスハイライト
- プラグイン機構
- テーマ切替
- クラウド同期
- 複数ユーザー共有を前提とした編集制御

## 7. 用語定義
- ホームフォルダ: NoCapEdit が新規ファイルを保存する基準フォルダ
- アクティブタブ: 現在ユーザーが表示・編集しているタブ
- 非アクティブ化: タブ切替などにより、そのタブが現在表示対象でなくなること
- 未保存フラグ: メモリ上の内容とディスク上の内容が一致していない状態を示す内部状態
- 自動生成ファイル: nce が起動時またはプラスボタン操作で新規作成したファイル
- セッション内作成フラグ: 現在のアプリ起動セッション中に作成されたファイルであることを示す内部状態
- 非空白入力履歴フラグ: そのファイルが一度でも空白以外の文字を含んだことがあるかを示す内部状態

## 8. 機能要件
### 8.1 初回起動設定
- 初回起動時にホームフォルダ設定画面を表示する
- デフォルトのホームフォルダは Windows の Documents 配下の nce フォルダとする
- 具体例: C:/Users/<ユーザー名>/Documents/nce
- ユーザーはデフォルト値をそのまま採用することも、変更することもできる
- 設定したホームフォルダは次回以降も保持する

### 8.2 起動時のファイル自動作成
- アプリ起動時、ホームフォルダ配下に新規テキストファイルを自動作成する
- ファイル名形式は yyyymmdd_hhmmss.txt とする
- 時刻はローカル時刻を用いる
- 生成されたファイルは作成後すぐにアクティブタブとして開く
- 同名ファイルが既に存在する場合は、_01、_02 のように連番を付けて回避する

### 8.3 タブ機能
- 複数ファイルをタブで開ける
- タブ名にはファイル名を表示する
- タブバーにプラスボタンを表示し、操作時に新規ファイルを作成して新規タブとして開く
- 初期MVPではタブ数の明示的な上限は設けない
- 各タブは少なくとも、ファイルパス、自動生成ファイルかどうか、セッション内作成フラグ、未保存フラグ、非空白入力履歴フラグを内部状態として保持できること

### 8.4 編集機能
- アクティブタブ内のテキストを編集できる
- 初期MVPはプレーンテキスト編集を対象とする
- 文字コードは UTF-8 とする
- Windows向け初期MVPの保存改行コードは CRLF とする

### 8.5 自動保存
- アクティブタブで編集内容が変更された場合、未保存フラグを立てる
- 編集内容に空白以外の文字が含まれた時点で、非空白入力履歴フラグを立てる
- 初期MVPの自動保存待機時間は 3 秒とする
- 入力が停止してから 3 秒経過した時点で、自動保存を実行する
- 自動保存は常時ポーリングではなく、入力イベント契機でタイマーを再設定するイベント駆動方式とする
- 将来的に自動保存待機時間は設定で変更可能にすることを想定し、実装上は固定値の差し替えがしやすい構成とする
- タブが非アクティブになった時点で、そのタブに未保存フラグがあれば保存する
- タブを閉じる時点で、そのタブが自動生成ファイルかつセッション内作成フラグが有効であり、内容が空または空白文字と改行のみで構成され、かつ非空白入力履歴フラグが未設定の場合は、そのファイルを削除して閉じる
- 上記の削除条件に当てはまらない場合、タブを閉じる時点でそのタブに未保存フラグがあれば保存する
- アプリ終了時に、各タブが自動生成ファイルかつセッション内作成フラグが有効であり、内容が空または空白文字と改行のみで構成され、かつ非空白入力履歴フラグが未設定の場合は、そのファイルを削除対象とする
- 上記の削除条件に当てはまらないタブについては、アプリ終了時に未保存フラグがある場合のみ保存を試行する
- 初期MVPでは全タブの定期一斉保存機能は実装しない

### 8.6 保存方式
- 保存は上書き保存を基本とする
- 安全性確保のため、一時ファイルに書き出してから置換する方式を採用する
- 保存成功時は未保存フラグを解除する
- 空白のみファイルの削除判定では、空文字、スペース、タブ、改行、およびそれらの組み合わせを空とみなす

### 8.7 保存失敗時の挙動
- 保存失敗時は、ユーザーに失敗が分かる形で通知する
- アプリ終了時の保存失敗では、少なくとも以下の選択肢を提示する
- 再試行
- 別名で保存
- 終了を中止
- タブ終了時の保存失敗では、原則としてタブを閉じず、ユーザーに再試行または別名保存の判断を委ねる

### 8.8 保存状態表示
- 画面下部にステータスバーを設ける
- ステータスバーには、少なくとも以下の状態を表示できるようにする
- 保存済み
- 編集中
- 保存中
- 保存失敗

## 9. UI要件
### 9.1 全体方針
- シンプルで落ち着いた見た目にする
- 装飾過多を避け、余白と境界線を活かしたモダンな構成にする
- Obsidian 風の雰囲気を参考にしつつ、NoCapEdit に必要な最小要素へ絞る

### 9.2 最低限必要な画面要素
- タブバー
- 新規タブ作成用のプラスボタン
- テキスト編集エリア
- 画面下部のステータスバー

### 9.3 初期表示
- 起動直後に自動生成されたファイルを開いた状態を表示する
- タブ名にはそのファイル名を表示する

## 10. データと設定
### 10.1 保存先
- 保存先はホームフォルダ配下とする
- 初期MVPでは新規自動作成ファイルはすべてホームフォルダ直下に作成する

### 10.2 設定保持
- ホームフォルダ設定は永続化する
- 初期MVPでは設定項目はホームフォルダを最優先とする
- 自動保存待機時間の設定変更機能はMVP対象外とするが、将来追加を想定する

## 11. 非機能要件
### 11.1 パフォーマンス
- 無操作時に継続的な入力監視を行わないこと
- 自動保存の待機はイベント駆動で実現し、常時高負荷にならないこと

### 11.2 信頼性
- 保存時にファイル破損リスクを下げるため、安全保存方式を採用すること
- アプリ終了時には未保存内容の取りこぼしを最小限に抑えること

### 11.3 拡張性
- 将来的な macOS、Linux、CLI 対応を妨げない構成にすること
- 既存ファイルを開く機能、設定拡張、テーマ変更などを後から追加しやすい構造にすること

## 12. ユーザーフロー
### 12.1 初回起動
1. ユーザーがアプリを起動する
2. ホームフォルダ設定画面が表示される
3. ユーザーがデフォルト値を承認、または変更する
4. 設定完了後、アプリが新規ファイルを作成して開く
5. ユーザーはすぐに入力を開始できる

### 12.2 通常利用
1. ユーザーがアプリを起動する
2. 新規ファイルが自動作成され、タブで開かれる
3. ユーザーが入力する
4. 入力停止から 3 秒後に自動保存される
5. 必要に応じてプラスボタンから追加タブを作成する
6. タブ切替時には、元のタブが未保存なら保存される

### 12.3 終了時
1. ユーザーがタブを閉じる、またはアプリ終了を実行する
2. 対象タブが自動生成ファイルで、当該セッション中に作成され、内容が空白のみで、かつ非空白入力履歴がなければ削除する
3. 上記以外で対象タブに未保存内容があれば保存を試行する
4. 保存成功または削除成功なら終了を継続する
5. 保存失敗なら再試行、別名保存、終了中止を選択できる

## 13. 受け入れ基準
- 初回起動時にホームフォルダ設定が行える
- 2回目以降の起動では設定済みホームフォルダを使用できる
- 起動時に新規 .txt ファイルが自動作成され、すぐ編集できる
- ファイル名が yyyymmdd_hhmmss.txt 形式で生成される
- 同名ファイル衝突時に連番回避できる
- プラスボタンから新規タブを追加できる
- 入力停止 3 秒後に自動保存される
- タブ切替時に未保存内容が保存される
- 自動生成された当該セッション内のファイルで、内容が空白のみかつ非空白入力履歴がない場合、タブ終了時に削除される
- タブ終了時に未保存内容が保存される
- 自動生成された当該セッション内のファイルで、内容が空白のみかつ非空白入力履歴がない場合、アプリ終了時に削除される
- アプリ終了時に未保存内容が保存される
- 保存失敗時に再試行、別名保存、終了中止ができる
- ステータスバーで保存状態を確認できる
