# 1-2. 同名ファイル自動回避ループの上限厳格化

フェーズ1-2の実装計画です。開発初期に設定された「とりあえずの余裕をもった上限」を見直し、フェイルファスト（早期異常検知）の原則に従って上限を狭めることで、バグによる意図しない大量のゴミファイル生成を防ぐ堅牢な設計を目指します。

## 背景と目的

現在、ファイル保存時や新規作成時に同名ファイルが存在する場合、`_01` から `_99` まで自動で連番を振るループ処理が実装されています。しかし、手動操作を前提とする本アプリにおいて、同一秒間にファイルが大量生成される状況は「システムの暴走（無限ループなど）」に他なりません。
上限が99の場合、暴走時に99個のゴミファイルが生成されてしまい、ユーザーのストレージを圧迫するリスクがあります。
そのため、上限を `9`（`_09` まで）へ厳格化し、異常発生時に被害を最小限に食い止めて早期にエラーを返すよう修正します。

## 提案する変更

### バックエンド修正
#### [MODIFY] [main.rs](file:///c:/work/NoCapEdit/src/main.rs)
- `next_available_file_path` 関数内でのインクリメントループの上限判定を `if index > 99` から `if index > 9` へ修正します。

```rust
        index += 1;
        if index > 9 { // 変更箇所
            return Err("同名ファイル回避の上限に達しました".to_string());
        }
```

※今回の修正は `next_available_file_path` に限定しています。コピーなどで使用される他のパス取得関数（もし存在する場合）は、また別の要件（同一名のファイルを複数回コピーするなど）があるため今回のスコープ外とします。
